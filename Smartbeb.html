買保險 SmartBeb - 精算自己的保單

Introduction

Fintech 近幾年蓬勃發展，從銀行證券到保險，越來越多金融產品數位化，現在市面上幾個大型的保險資訊平台網站，買保險可以說佔了其中一個重要的席次。在使用者與口碑增加之後，買保險開始思考如何讓自己能夠在業務的銷售流程中獲得重要的地位，我加入專案時正是開始從零規劃的時候。

為遵守保密協議，某些數據與研究會被省略或隱藏

Design Process

Goals

法律上買保險依然需要與保險員進行面對面的簽約流程，整個流程設計上將會從網路行銷到網頁保單，該網站原本就有提供許多保單的知識與交流文章，最後鼓勵消費者自行計算後與保險員聯絡。

Features and structure

Planning

根據需求，我們知道所要做的東西共有三項：優化保險推薦、新增商品組合、新增商品搜尋，後兩項是重新規劃的功能，並要與前面的保險推薦進行連結。至於保險推薦將來要因應行銷活動以及成為客戶來到買保險網站的首要選擇之一，所以我們就先從現有的保險推薦起頭，當作我們流程的第一站。

Usability Testing

Alpha test

我們召集了專門處理線上保單的業務部門，讓他們嘗試使用我們的功能，事實上業務更專注於保單內容的正確性，而我針對他們是否能夠正確找到習慣的內容以及抓取資料的時間長度來判斷他們是否能夠在介面上模擬他們與真實客戶應對的內容。

Workshop

內部測試一陣子後，為了能夠更了解業務的實際狀況，我們在業務的銷售流程中取出四個主要階段：開發客戶、交涉內容、如何成交與售後服務，並讓他們在每個階段寫下他們最常見的狀況與會遇到的的問題，讓他們針對我們的產品提出可以改善的意見。

A/B test

我們在上述的測試中發現到了幾個問題，我們測試了幾個項目，其中一個項目（如圖）是位於保險推薦內的首次進入時會出現的保險分類內容，原本希望藉由縮小客戶的需求來達到聚焦推薦的商品，但我們發現高達百分之九十的人都會右上角點掉，並且也接到內部測試訊息覺得這個功能更加誤導，因而最後經過測試與遊說，我們把它刪除了，並且讓有更多人往後續的流程移動。

Release

經過幾次測試，我們根據他們的問題以及我們在頁面上操事的結果，將好與不好的內容增加或減少進去，之後就是正式的發表上線的消息了。
