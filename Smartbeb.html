買保險 SmartBeb - 精算自己的保單

Introduction

Fintech 近幾年蓬勃發展，從銀行證券到保險，越來越多金融產品數位化，現在市面上幾個大型的保險資訊平台網站，買保險可以說佔了其中一個重要的席次。在使用者與口碑增加之後，買保險開始思考如何讓自己能夠在業務的銷售流程中獲得重要的地位，我加入專案時正是開始從零規劃的時候。

為遵守保密協議，某些數據與研究會被省略或隱藏

Design Process

Goals

法律上買保險依然需要與保險員進行面對面的簽約流程，整個流程設計上將會從網路行銷到網頁保單，該網站原本就有提供許多保單的知識與交流文章，最後鼓勵消費者自行計算後與保險員聯絡。

Features and structure

Planning

根據需求，我們知道所要做的東西共有三項：優化保險推薦、新增商品組合、新增商品搜尋，後兩項是重新規劃的功能，並要與前面的保險推薦進行連結。至於保險推薦將來要因應行銷活動以及成為客戶來到買保險網站的首要選擇之一，所以我們就先從現有的保險推薦起頭，當作我們流程的第一站。

Structure

三大工具規劃
  1. 保險推薦 - 主要使用者為民眾，推薦網站上已經組好的保單組合。
  2. 保險查詢 - 可搜尋目前網站上所有歸檔在案的保單。
  3. 保險組合器 - 主要使用者為業務，以自由組合保單功能為賣點。

三個工具互通有無，最終會因應業務的銷售流程串連這三個工具的所有內容。

三大工具的簡易流程圖

主要能讓三大工具的最後流程能夠互相通達，最後到我們希望的目的地：留單。

Design

我們製作中保真的 Mockup 來跟 PM 進行溝通。將必要欄位與必要流程加入其中，利用視覺解決那些只有文字與口述的內容。

Usability Testing

Alpha test

我們召集了專門處理線上保單的業務部門，讓他們嘗試使用我們的功能，事實上業務更專注於保單內容的正確性，而我針對他們是否能夠正確找到習慣的內容以及抓取資料的時間長度來判斷他們是否能夠在介面上模擬他們與真實客戶應對的內容。

Workshop

內部測試一陣子後，為了能夠更了解業務的實際狀況，我們在業務的銷售流程中取出四個主要階段：開發客戶、交涉內容、如何成交與售後服務，並讓他們在每個階段寫下他們最常見的狀況與會遇到的的問題，讓他們針對我們的產品提出可以改善的意見。

A/B test

我們在上述的測試中發現到了幾個問題，我們測試了幾個項目，其中一個項目（如圖）是位於保險推薦內的首次進入時會出現的保險分類內容，原本希望藉由縮小客戶的需求來達到聚焦推薦的商品，但我們發現高達百分之九十的人都會右上角點掉，更接到內部測試訊息覺得這個功能更加誤導。

通過 A/B Test 以及八位的面對面訪談人員後，得出幾個結論：

  1. 情境彈跳視窗沒有特別的幫助，只是會干擾想要計算條件的使用者。
  2. 許多人點完情境之後，無法理解後續的流程與他的相關性。
  3. 取消情境視窗後，往後續流程的人比較多。

最終我們把它刪除了＋加強底下三組原有推薦保險組合的視覺效果，讓客戶能夠注意到我們推薦給他的選項，讓有更多人往後續的流程移動。

Release

經過幾次測試，我們根據他們的問題以及我們在頁面上操事的結果，將好與不好的內容增加或減少進去，之後就是正式的發表上線的消息了。

1.Insurance Recommend Page

由專業的人員配置，使用者輸入自己的條件可以算出自己在該組合保單的費用。推薦就是懶人包，通常是市面上最受歡迎的幾種配方。客戶不可變動、行銷功能強大。

  • 小白
  • 一知半解

2.Search Page

使用者可以搜尋自己得知的保單，甚至可以得知以前保單的版本。預計做成保險界的 GOOGLE ，搜尋的部分多半著重在資料庫的豐富性，視覺上清楚清晰就好了。（這也是我第一次做到這麼多資訊的搜尋結果列表...）

  • 小白
  • 一知半解
  • 資優生

3.Combination set Page

除了提供組合自己的保險之外，底下還提供險種比例分析、理賠項目列表以及逐年保費表格與圖表提供參考。給資優生與業務的頁面，讓這些保險知識達人們可以做客製化的地方，可以將自己做好的保單分享給別人，每個保險商品項目也可以連結到自己的資訊頁面，是與其他頁面最流通的地方。

  • 資優生

What's next?

將三大工具製作出之後，原本的優化計畫會繼續進行。我與PM則要繼續計畫基於這三個功能衍伸出來的內容，包括保險商品推薦行銷、退休計畫試算、公司本身的特別產品等等，開始奠定保險電子化的基礎。
